================================================================================
  SLOTIFY — SALON SLOT BOOKING SYSTEM
  Full Project Explanation (Inch by Inch)
================================================================================

--------------------------------------------------------------------------------
WHAT IS THIS PROJECT?
--------------------------------------------------------------------------------
Slotify is a full-stack web app for a salon/barbershop.
It lets customers browse services, book time slots, pay online, and get email
confirmations — while giving the admin a full control panel to manage bookings,
revenue, and services.

--------------------------------------------------------------------------------
HIGH-LEVEL ARCHITECTURE
--------------------------------------------------------------------------------

  salon-slot-booking/
  ├── client/     ← Next.js Frontend (React)
  ├── server/     ← Node.js + Express Backend
  ├── .htaccess   ← Apache routing rules (for deployment on shared hosting)
  └── README.md

The project is split into TWO separate apps — a frontend and a backend —
that talk to each other via a REST API.

================================================================================
  BACKEND — server/ (Node.js + Express)
================================================================================

--------------------------------------------------------------------------------
Entry Points
--------------------------------------------------------------------------------

  File        | Purpose
  ------------|----------------------------------------------------------------
  server.js   | Starts the HTTP server, connects to MongoDB, launches
              | schedulers (reminder emails & self-ping to prevent cold starts)
  app.js      | Creates the Express app, registers all middleware and routes
  seed.js     | One-time script to pre-populate the DB with sample services

--------------------------------------------------------------------------------
server/models/ — Database Schemas (MongoDB via Mongoose)
--------------------------------------------------------------------------------
These define the SHAPE OF THE DATA stored in the database.

  File          | Collection  | Key Fields
  --------------|-------------|------------------------------------------------
  User.js       | users       | name, email, password (bcrypt hashed), phone,
                |             | role (customer/admin), totalBookings, totalSpent
  Booking.js    | bookings    | user (ref), service (ref), date (YYYY-MM-DD),
                |             | slotTime (HH:mm), status (pending/confirmed/
                |             | completed/cancelled/expired/blocked),
                |             | reminderSent
  Service.js    | services    | name, description, price, duration (in mins)
  Payment.js    | payments    | booking (ref), user (ref), amount, status,
                |             | paymentMethod

--------------------------------------------------------------------------------
server/routes/ — URL Routing
--------------------------------------------------------------------------------
Each file maps URL paths to controller functions.

  File                 | Base URL          | Endpoints
  ---------------------|-------------------|-------------------------------------
  authRoutes.js        | /api/auth         | POST /register, POST /login
  bookingRoutes.js     | /api/bookings     | GET all, POST create, PUT edit/
                       |                   | cancel, DELETE, GET available slots
  serviceRoutes.js     | /api/services     | GET all, POST/PUT/DELETE (admin only)
  paymentRoutes.js     | /api/payments     | POST create payment, GET history
  reportRoutes.js      | /api/reports      | GET revenue summary, booking stats

--------------------------------------------------------------------------------
server/controllers/ — Business Logic
--------------------------------------------------------------------------------
The "brains" — each controller handles what actually happens when a route is hit.

  File                    | Key Functions
  ------------------------|------------------------------------------------------
  authController.js       | registerUser() — creates user
                          | loginUser() — validates password, returns JWT token
  bookingController.js    | createBooking() — check slot availability → create
                          | booking → send confirmation email
                          | editBooking(), cancelBooking(), blockSlot() (admin),
                          | getAvailableSlots()
  serviceController.js    | getServices(), createService(),
                          | updateService(), deleteService()
  paymentController.js    | createPayment() — records payment, updates totalSpent
                          | on user; getPayments()
  reportController.js     | getRevenueReport() — aggregates booking/payment data
                          | for admin charts

--------------------------------------------------------------------------------
server/middleware/ — Request Interceptors
--------------------------------------------------------------------------------

  File                  | What it does
  ----------------------|--------------------------------------------------------
  authMiddleware.js     | protect() — reads JWT token from request header,
                        | verifies it, attaches req.user to the request
  adminMiddleware.js    | adminOnly() — runs after protect(), checks if
                        | req.user.role === 'admin', else returns 403 Forbidden
  errorMiddleware.js    | Global error handler — catches all errors and sends
                        | consistent JSON error responses

--------------------------------------------------------------------------------
server/services/ — Background Services
--------------------------------------------------------------------------------

  File                  | What it does
  ----------------------|--------------------------------------------------------
  emailService.js       | Sends emails via Resend API — booking confirmations,
                        | reminders, and cancellation notices
  reminderService.js    | Runs on a cron schedule — finds bookings due tomorrow
                        | with reminderSent: false, emails all of them
  revenueService.js     | Helper functions for computing daily/monthly revenue
                        | aggregations

--------------------------------------------------------------------------------
server/schedulers/ — Cron Jobs
--------------------------------------------------------------------------------
Timed background tasks:
  - Mark expired bookings automatically
  - Send reminder emails the day before an appointment
  - Self-ping the Render backend every 14 minutes to prevent cold starts

--------------------------------------------------------------------------------
server/config/ — Database Connection
--------------------------------------------------------------------------------
Contains db.js — the Mongoose connect() call using MONGO_URI from the .env file.

================================================================================
  HOW A BOOKING WORKS (Full Flow)
================================================================================

  Customer → /services        → picks a service
           → /book-slot       → picks a date & time slot
           → API POST /api/bookings
           → server checks slot availability
           → Payment recorded
           → Booking created in MongoDB
           → emailService sends confirmation email (via Resend)
           → reminderService emails again the day before the appointment

================================================================================
  AUTHENTICATION FLOW
================================================================================

  Register/Login → server hashes password (bcrypt)
                → returns JWT token
  Client stores token in localStorage via AuthContext
  Every API request → Axios sends token in Authorization header
  authMiddleware verifies token → grants or denies access
  Admin routes also run adminMiddleware → checks role === 'admin'

--------------------------------------------------------------------------------
HOW SUPER ADMIN ACCESS CONTROL WORKS
--------------------------------------------------------------------------------

  1. Super admin logs in at /admin/login with their credentials
  2. Backend returns user object with role: 'superadmin'
  3. AuthContext stores this user in state
  4. Admin layout reads user.role — if 'superadmin', renders the
     "Manage Admins" nav item; otherwise it is simply not rendered
  5. The Manage Admins page double-checks on mount and redirects away
     if user is not superadmin
  6. All /api/admin/* API calls also verify role server-side via
     superAdminOnly middleware — even if someone manually navigates there

  Summary: Access is enforced at BOTH the UI layer and the API layer.

  AI STYLING FEATURE (NEW 🔥)

--------------------------------------------------------------------------------
OVERVIEW
--------------------------------------------------------------------------------
The AI Style Analyzer allows users to upload a selfie and receive personalized
hairstyles and beard suggestions based on their face shape.

  Flow:
    1. User uploads a selfie on the /ai-style page.
    2. Backend sends the image to OpenAI Vision API (GPT-4o).
    3. AI analyzes face shape and returns suggestions in JSON.
    4. Backend maps suggestions to existing Services in the Database.
    5. UI displays results with direct "Book Now" links.
